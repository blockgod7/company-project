# Drawing Management UI Interview Context

Date: 2026-07-02

## Trigger

User wants a deep interview for the drawing management tab UI, using an Autodesk Vault Professional screenshot as the reference.

## Reference Observation

The referenced Vault-like layout has:

- Left navigation tree showing folder structure at a glance.
- Center/top file table for the selected folder.
- File columns such as file name, extension, creator, drawing number, revision, state, category.
- Bottom preview/detail area for the selected file.
- Right properties panel with metadata.
- Top command/search area.

## Current Product Spec Context

Phase 1 drawing management already defines:

- Product drawings organized by company/customer/vendor.
- Equipment drawings organized by equipment and common groups/folders.
- Uploaded files are stored in the system.
- Latest version is metadata-driven.
- Drawing number is globally unique.
- Duplicate drawing number on new upload should warn and guide to revision/cancel.
- View and download are permission-controlled.
- Downloads require approval and are valid for 4 hours after approval.

## Current Implementation Shape

The current React page is tab/form/list oriented:

- Tabs: drawings, downloads, permissions.
- Drawing list and upload form appear side by side.
- Detail panel shows revisions, revision upload, and download request controls.
- Admin permissions are handled in a separate tab.

This works functionally but is not yet a Vault-like drawing workbench.

## UI Interview Focus

Primary ambiguity:

- Whether the default drawing management screen should become a persistent Vault-style workbench with folder tree, file grid, preview/detail, and properties panels.
- Or whether Phase 1 should use a lighter left-tree + center-list + bottom-preview layout and move properties/admin forms into drawers/modals.

Potential UI zones to decide:

1. Left explorer tree.
2. Center file grid.
3. Bottom preview/detail tabs.
4. Right properties panel.
5. Top command/search bar.

## Interview Decisions

- Default drawing management layout should be: left tree + center list + bottom preview.
- The top area should include a file registration button.
- The right persistent properties panel is not part of the Phase 1 default layout.
- Drawing details/properties can move into the bottom area or an on-demand detail view.
- Product drawing tree should be: company/vendor -> project/product name -> drawings.
- Equipment drawing tree candidate should be: business unit -> process -> common drawing folder -> equipment folders -> drawings.
- Equipment drawing tree decision: common drawings and equipment folders should be siblings under each process.

## Interview Notes

- Product tree with project/product name is a strong fit because product drawings are usually searched by supplier/customer context and product/project context.
- Equipment tree needs one more decision: whether common drawings belong as a sibling of equipment folders under each process, or as a parent/group before equipment folders. This affects whether selecting a process shows both common and equipment drawings cleanly.
- Confirmed with user: common drawings as a sibling of equipment folders is the intended model.

## Current Tree Model

Product drawings:

```text
제품도면
└─ 업체명
   └─ 프로젝트/제품명
      └─ 도면
```

Equipment drawings:

```text
설비도면
└─ 사업부
   └─ 공정
      ├─ 공통도면
      ├─ 설비명
      └─ 설비명
```

## Center List Decision

The center drawing list should stay focused because the tree already provides the company/project/equipment context.

Initial columns:

- File name.
- Extension.
- Drawing number.
- Drawing title/name.
- Revision.
- Status.
- Creator or uploader.
- Registration date.

Context columns such as company, project, equipment, and common group do not need to be always visible in Phase 1.

## Bottom Preview Decision

Bottom area should use four tabs:

- Preview.
- Revision history.
- Download request history.
- Properties.

Preview behavior:

- PDF files should show an actual preview when technically available.
- CAD files should show a summary preview instead of CAD rendering in Phase 1.
- Summary preview should include file extension, file name, drawing number, latest revision, status, download request action, and simple notes/context.

## Top Bar Decision

The top area should support three primary jobs:

- File registration button on the left.
- Current selected path/breadcrumb in the center.
- Integrated search on the right.

Search behavior:

- Search fields: drawing number, file name, drawing title, revision, uploader/registrant.
- Default search scope should be the currently selected tree node.
- Provide an option to search all drawings when needed.
- File registration should prefill context from the selected tree node when possible.

## Latest Version Visibility Discussion

User wants the default folder view to show latest drawings first/only, to reduce the risk of users accidentally using lower revisions.

Recommended direction:

- Default center list should show only the latest active revision for each drawing number.
- Older revisions should not appear as normal peer files in the main list.
- Older revisions should remain accessible through a clear revision/history view.
- A virtual "(구)버전" or "이전버전" area may be useful, but it should be system-generated rather than a manually managed folder, so users cannot accidentally move or classify versions incorrectly.
- Decision: older revisions should only be visible in the bottom revision history tab. Do not show an old-version folder in the left tree in Phase 1.

## Revision Order Decision

- Revision label is user-entered/displayed.
- Revision order should be system-generated to prevent errors.
- First registration should default to order 1.
- Additional revisions should automatically receive the next order.
- Users should not normally edit revision order in the Phase 1 UI.

## Ambiguity Estimate

Current UI ambiguity: 4%.

Need to reduce below the standard interview threshold before implementation.
